# 04 詳細設計: AI プロンプト

## この工程の目的

基本設計を基に、**コーディング可能な粒度のクラス設計・処理フロー**を設計する工程です。
AI は基本設計書から、Pydantic モデル定義、サービスクラスの処理フロー、シーケンス図を自動生成します。

## AI への指示（プロンプト）

````
あなたは Python (FastAPI) での Web API 開発経験が 10 年以上ある
シニアエンジニアです。Clean Architecture とレイヤードアーキテクチャに精通しています。

@basic_design.md（基本設計書）を基に、詳細設計書を作成してください。
この設計書を読んだエンジニアが、追加の質問なしにコーディングを
開始できる粒度で記述してください。

## 作業手順（この順番で設計してください）

1. 基本設計書のテーブル定義・API 設計を読み込む
2. ディレクトリ構成を定義し、各ファイルの責務を明記する
3. 基本設計の API リクエスト/レスポンスの JSON から
   Pydantic v2 のスキーマ定義を Python コードとして生成する
4. 各サービスクラスのメソッドを設計し、処理フローを疑似コードで記述する
5. 主要なユースケースのシーケンス図を Mermaid で作成する
6. エラーコード一覧と例外クラスの階層を設計する

## 出力する詳細設計書の構成

### 1. ディレクトリ構成

以下のレイヤード構成に従うこと:
```
app/
├── main.py                      # エントリーポイント
├── core/                        # 設定・例外（横断的関心事）
├── models/                      # SQLAlchemy モデル（DB テーブル定義）
├── schemas/                     # Pydantic スキーマ（入出力定義）
├── services/                    # ビジネスロジック
├── repositories/                # DB 操作（SQL 発行）
├── api/v1/                      # FastAPI エンドポイント（Router）
├── db/                          # DB セッション管理
└── tests/                       # テストコード
    ├── unit/                    # サービス単位のテスト
    └── integration/             # API エンドポイント貫通テスト
```
各ファイルの責務を 1 行で説明すること。

### 2. Pydantic スキーマ定義

Python コードとして以下のルールで記述すること:
- Pydantic v2（BaseModel を継承）
- Field() でバリデーションルールを明記
- pattern= で文字列フォーマットを制約（例: ロット番号は ^\d{8}-[A-Z]+-\d{3}$）
- 必須フィールドと任意フィールドを明確に区別（Optional 使用）
- model_config = {"from_attributes": True} を設定（ORM モード）

以下のスキーマを必ず定義すること:
- InspectionCreate（検査結果登録リクエスト）
- InspectionDetailCreate（検査明細リクエスト）
- InspectionResponse（検査結果レスポンス）
- InspectionDetailResponse（検査明細レスポンス）
- LotCreate（ロット登録リクエスト）
- LotResponse（ロットレスポンス）
- DefectRateResponse（不良率集計レスポンス）
- QualityAlertResponse（品質アラートレスポンス）

### 3. サービスクラス設計

各サービスクラスのメソッドを以下の形式で記述すること:

```python
class InspectionService:
    def create_inspection(self, data: InspectionCreate) -> InspectionResponse:
        """検査結果を登録し、合否を自動判定する。

        処理フロー:
        1. ロット番号の存在チェック（→ NotFoundException）
        2. 重複チェック: 同一ロット・同一工程（→ DuplicateException）
        3. 検査基準の取得
        4. 各項目の合否判定: lower_limit <= measured_value <= upper_limit
        5. 全体合否判定: 1 項目でも FAIL なら全体も FAIL
        6. DB 保存
        7. FAIL の場合、AlertService.check_and_trigger_alert() を呼出
        """
```

### 4. 処理シーケンス

以下の 2 つのユースケースのシーケンス図を Mermaid sequenceDiagram で記述:
- 検査結果登録（タブレット → API → Service → Repository → DB → アラート判定）
- ダッシュボード表示（PC → API → Service → DB → 集計結果返却）

### 5. エラーハンドリング設計

エラーコード一覧を以下の表形式で記述:
| エラーコード | HTTPステータス | 説明 | 発生条件 |

例外クラスの階層を Python コードで記述:
- AppException（基底）
- NotFoundException（404）
- DuplicateException（409）

## やってはいけないこと

- 処理フローを「実装に委ねる」と省略しないこと（全ステップを書く）
- Pydantic スキーマに型ヒントを付けないまま定義しないこと
- サービスクラスと API エンドポイントにビジネスロジックを混在させないこと
  （ビジネスロジックは必ずサービス層に集約）
- シーケンス図で「処理」とだけ書いて詳細を省略しないこと

## 品質チェックリスト（出力前に自己確認すること）

- [ ] 基本設計の全 API エンドポイントに対応するスキーマが定義されているか
- [ ] 全 Pydantic スキーマに型ヒントと Field バリデーションがあるか
- [ ] サービスクラスの処理フローが番号付きステップで記述されているか
- [ ] シーケンス図がタブレット/PCからDBまでの全レイヤーを含んでいるか
- [ ] エラーコード一覧が詳細設計の全エラーケースをカバーしているか
````

## AI 活用のポイント

| 観点 | 従来の方法 | AI 活用後 | 改善効果 |
|------|-----------|----------|---------|
| 所要時間 | 設計者が1〜2週間かけてクラス図・シーケンス図を作成 | AIが数分で生成 | **90%以上の時間削減** |
| Pydantic定義 | 基本設計のJSON例から手作業で変換 | 自動変換 | 転記ミス防止 |
| 処理フロー | 記述漏れが発生しがち | AIが網羅的にフローを生成 | 抜け漏れ防止 |

## 人間の役割

1. **設計の整合性確認**: 基本設計との乖離がないかレビュー
2. **パフォーマンスの考慮**: N+1問題等の性能懸念をチェック
3. **エッジケースの補完**: AI が見落とす業務特有のケースを追加

```
          │
```

````
## 追加するテストケース

### 追加正常系（TestAdditionalSuccess）

1. 短時間勤務（4時間: 9:00-13:00）→ 実働4h、休憩0h、残業0h
   - 6時間以下なので休憩控除なしを確認

2. ちょうど6時間超の勤務（9:00-15:30、拘束6.5h）→ 休憩45分控除、実働5.75h
   - 6時間超8時間以下なので休憩45分を確認

3. フレックス制: 月の総実労働が所定時間ちょうど → 残業0h
   - 総実労働160h、所定労働日数20日 → 残業0h

4. フレックス制: 月の総実労働が所定時間未満 → 残業0h（マイナスにならない）
   - 総実労働150h、所定労働日数20日 → 残業0h

### 追加境界値（TestAdditionalBoundary）

5. 拘束時間ちょうど8時間（9:00-17:00）→ 休憩45分控除、実働7.25h、残業0h
   - 拘束8時間は「8時間超」ではないので休憩は45分のはず
   - ただし、8時間ちょうどは「超える」に含まないため、休憩45分が正しい

6. 拘束時間8時間1分（9:00-17:01）→ 休憩60分控除
   - 8時間を超えたので休憩60分に変わる
````


### 5. テストが失敗した場合

失敗したテストのエラーメッセージを Gemini CLI に貼り付けて対応を依頼します:

```
以下のテストが FAIL しています。
テストの期待値が正しいか、実装のロジックが正しいかを分析して、
適切な方を修正してください:

（エラーメッセージを貼り付け）
```

> **ポイント**: 追加テストでは「テストの期待値自体が正しいか」の判断も必要です。
> 特に休憩控除の境界（8時間ちょうど = 45分 or 60分）は、
> 労基法の「超える」の解釈に依存するため、業務的な判断が求められます。

## この工程で学ぶこと

| 学習ポイント | 説明 |
|------------|------|
| V字の対応関係 | 詳細設計のメソッド仕様 → 単体テストで検証、という対応が明確 |
| テスト追加のワークフロー | 既存テストを壊さずに新テストを追加する手法 |
| 境界値テストの難しさ | 「ちょうど8時間」の休憩は45分か60分か -- 法律の解釈が必要 |
| AI のテスト生成力 | 具体的なテストケースを指示すれば、AI は正確なテストコードを生成する |

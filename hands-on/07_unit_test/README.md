# Step 07: 単体テスト

## この工程の位置づけ

```
      基本設計 ────────────────────── 結合テスト
        │                              │
        詳細設計 ──────── >>> 単体テスト <<<
          │
          コーディング ── コードレビュー
```

詳細設計で定義したクラス・メソッドが**設計通りに動作するか**を検証します。
Step 05 で TDD 用に書いた 13件のテストに加え、**追加のテストケース**を生成してカバレッジを向上させます。

## やること

Step 05 で作成した `AttendanceService` のテストに加え、AI が自律的に追加テストを生成・実行・修正します。

### Phase 1: 追加テストの生成と保存

既存のテスト（13件）はそのまま残し、以下の新しいテストケースを追加した
完全なファイルで `tests/test_attendance_service.py` を上書き保存してください。

#### 追加するテストケース

##### 追加正常系（TestAdditionalSuccess）

1. 短時間勤務（4時間: 9:00-13:00）→ 実働4h、休憩0h、残業0h
   - 6時間以下なので休憩控除なしを確認

2. ちょうど6時間超の勤務（9:00-15:30、拘束6.5h）→ 休憩45分控除、実働5.75h
   - 6時間超8時間以下なので休憩45分を確認

3. フレックス制: 月の総実労働が所定時間ちょうど → 残業0h
   - 総実労働160h、所定労働日数20日 → 残業0h

4. フレックス制: 月の総実労働が所定時間未満 → 残業0h（マイナスにならない）
   - 総実労働150h、所定労働日数20日 → 残業0h

##### 追加境界値（TestAdditionalBoundary）

5. 拘束時間ちょうど8時間（9:00-17:00）→ 休憩45分控除、実働7.25h、残業0h
   - 拘束8時間は「8時間超」ではないので休憩は45分

6. 拘束時間8時間1分（9:00-17:01）→ 休憩60分控除
   - 8時間を超えたので休憩60分に変わる

#### テストコードのルール

- 既存テストと同じ fixture（db_session, sample_employee）を使用
- 各テストに日本語 Docstring
- assert は具体的な Decimal 値で比較
- 既存の TestCalculateDailySummarySuccess、TestAttendanceErrors、TestWorkingHoursBoundary は変更しない

---

### Phase 2: テスト実行と修正

テストファイルを保存したら、以下を実行:

```bash
python -m pytest tests/test_attendance_service.py -v
```

テストが失敗した場合は:
1. テストの期待値が正しいか、実装のロジックが正しいかを分析する
2. 適切な方を修正する（特に休憩控除の境界は労基法の「超える」の解釈に注意）
3. 再度テストを実行する

**全テストが PASSED になるまでこのサイクルを繰り返してください。**

## この工程で学ぶこと

| 学習ポイント | 説明 |
|------------|------|
| V字の対応関係 | 詳細設計のメソッド仕様 → 単体テストで検証、という対応が明確 |
| AI の自律テスト | テスト追加 → 実行 → 失敗分析 → 修正のサイクルを AI が自律的に回す |
| 境界値テストの難しさ | 「ちょうど8時間」の休憩は45分か60分か -- 法律の解釈が必要 |
| 既存テストとの共存 | 既存テストを壊さずに新テストを追加する手法 |

````
## レビュー観点（7つ全てを必ず実施すること）

### 1. 仕様整合性
- 勤務時間の計算ロジックが正しいか
- 休憩時間の自動控除が労基法34条に準拠しているか（6h超→45分、8h超→60分）

### 2. セキュリティ
- SQL インジェクションの危険性
- 個人情報（位置情報含む）の取り扱い

### 3. パフォーマンス
- N+1 問題
- 不要なクエリ

### 4. エラーハンドリング
- 例外の握りつぶし（except: pass）
- トランザクション制御の不備

### 5. 型安全性
- 型ヒントの欠落
- Any 型の乱用

### 6. 命名規約
- 命名の一貫性（snake_case）

### 7. アーキテクチャ
- Controller 層にビジネスロジックが漏れていないか
- サービス層の責務分離

## 出力形式

### 指摘一覧

| # | ファイル | 行番号 | 深刻度 | カテゴリ | 指摘内容 | 修正案 |
|---|---------|--------|--------|---------|---------|--------|

深刻度の定義:
- **Critical**: セキュリティ脆弱性、データ破損の可能性 → 必ず修正
- **Major**: パフォーマンス問題、ロジックバグ → 修正推奨
- **Minor**: 命名規約違反、コードスタイル → 修正任意

### サマリー

- Critical: X件
- Major: X件
- Minor: X件
- 総合評価: （合格 / 条件付き合格 / 要修正）

## やってはいけないこと

- 全て「問題なし」で終わらせないこと（最低 5 件は指摘すること）
- 修正案なしの指摘をしないこと（必ず具体的な修正コードを示す）
- 深刻度を全て Minor にしないこと

@app/services/attendance_service.py
@app/models/attendance.py
@app/models/employee.py
@app/core/exceptions.py
````

### 3. レビュー結果を確認

- [ ] Critical の指摘がある場合は、必ず修正する
- [ ] Major の指摘は、業務判断して修正するかどうかを決定する
- [ ] 修正案のコードが妥当かを確認する

### 4. 指摘事項を修正（任意）

Critical/Major の指摘がある場合は、Gemini CLI に修正を依頼します:

```
上記のレビューで Critical と Major の指摘を全て修正した
コードを出力してください。修正箇所にはコメントで
「# REVIEW FIX: 指摘内容」を付けてください。
```

修正後、再度テストを実行して PASS することを確認してください:

```bash
python -m pytest tests/test_attendance_service.py -v
```

### 5. レビュー結果を保存


## 期待される出力の例

一般的に以下のような指摘が検出されます:

- **Critical**: 深夜時間計算で日跨ぎケースの考慮漏れ
- **Major**: Service層でcommit()しておりトランザクション管理が不適切
- **Major**: 打刻の競合状態（Race Condition）への対策不足
- **Minor**: Decimal変換の精度に関する改善提案
- **Minor**: _validate_employee の冗長性

## この工程で学ぶこと

| 学習ポイント | 説明 |
|------------|------|
| AI レビューは網羅的 | 7つの観点を漏れなくチェックでき、人間のレビューの補完になる |
| 深刻度の分類が重要 | Critical/Major/Minor の分類で、修正の優先度を判断できる |
| 修正案の具体性 | 「こうすべき」だけでなく「こう書き換える」という具体的なコードを出力させる |
| AI レビューの限界 | ビジネスロジックの妥当性（労基法の正確性など）は人間の判断が必要 |

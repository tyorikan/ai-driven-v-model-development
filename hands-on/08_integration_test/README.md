# Step 08: 結合テスト

## この工程の位置づけ

```
    要件定義 ────────────────────────── システムテスト
      │                                    │
      基本設計 ──────────────── >>> 結合テスト <<<
        │
        詳細設計 ──────── 単体テスト
```

基本設計で定義した **API エンドポイント間の連携**が正しく動作するかを検証します。
単体テストが「メソッド単位」の検証であるのに対し、結合テストは**「API を通じたシナリオ」**で検証します。

> **前提**: Step 05 で API 層（`app/main.py`, `app/api/v1/attendance.py`）と
> 結合テスト（`tests/test_api_integration.py`）が既に生成されています。
> この Step では、追加の結合テストシナリオを生成してカバレッジを強化します。

---

## やること

Step 05 で生成した結合テストに加え、AI が自律的に追加シナリオを生成・実行・修正します。

### Phase 1: 追加結合テストの生成と保存

既存の結合テストはそのまま残し、以下の追加テストシナリオを含めた
完全なファイルで `tests/test_api_integration.py` を上書き保存してください。

#### 追加テストシナリオ

##### シナリオ3: 境界値の API レベル検証（TestApiBoundaryScenarios）

1. 定時勤務（9:00-18:00）で打刻 → 日次集計 API で残業0h を確認
2. 残業勤務（9:00-20:00）で打刻 → 日次集計 API で残業2h を確認
3. 深夜勤務（9:00-23:30）で打刻 → 日次集計 API で深夜1.5h を確認

##### シナリオ4: エラーレスポンスの検証（TestApiErrorResponses）

4. 各エラーレスポンスに error_code フィールドが含まれることを検証
   - 404: error_code = "EMPLOYEE_NOT_FOUND"
   - 409: error_code = "DUPLICATE_CLOCK_IN"
   - 400: error_code = "VALIDATION_ERROR"

5. エラーレスポンスのボディ形式が統一されていることを確認
   - { "error_code": "...", "detail": "..." } 形式

#### テストコードのルール

- 既存の fixtures を再利用
- 各テストに日本語 Docstring
- レスポンスの status_code と body の両方を検証

---

### Phase 2: テスト実行と修正

テストファイルを保存したら、全テストを実行:

```bash
python -m pytest tests/ -v
```

テストが失敗した場合は:
1. API のエンドポイント実装とテストの両方を確認する
2. 適切な方を修正する
3. 再テストする

**全テストが PASSED になるまでこのサイクルを繰り返してください。**

## この工程で学ぶこと

| 学習ポイント | 説明 |
|------------|------|
| V字の対応関係 | 基本設計の API → 結合テストで検証、という対応が明確 |
| シナリオベースのテスト | 単一 API ではなく、API の連携（打刻→集計）を検証する |
| エラーレスポンスの統一 | error_code を含む統一形式のエラーレスポンスを検証する |
| AI の自律デバッグ | API 層とテストの不整合を AI が自律的に分析・修正する |

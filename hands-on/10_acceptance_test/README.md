# Step 10: 受け入れテスト

## この工程の位置づけ

```
  >>> 要件分析 <<< ────────────────────── >>> 受け入れテスト <<<
    │                                          │
    要件定義 ────────────────────────── システムテスト
```

V字モデルの最上流（要件分析）と最下流（受け入れテスト）が対応する、**最終検証工程**です。
RFP の全ての要求が実現されているかを、**トレーサビリティマトリクス**で検証します。

RFP と要件定義書を基に、AI が自律的に**受け入れテスト仕様書**を生成・保存・レビュー・修正します。

### Phase 1: 成果物の生成と保存

以下の作業手順と出力形式に従って受け入れテスト仕様書を作成し、
`acceptance_test_plan.md` に保存してください。

#### 作業手順

1. RFP のセクション5「希望する機能」の全項目を抽出する
2. 各機能に対応する要件ID（F-01〜）を紐付ける
3. 各要件に対するテストケースを設計する
4. トレーサビリティマトリクスで全要求のカバレッジを確認する

#### 出力形式

##### 1. トレーサビリティマトリクス

RFP の要求 → 要件ID → テストケースID の対応表:

| RFP セクション | RFP の要求内容 | 要件ID | テストケースID | カバレッジ |
|--------------|--------------|--------|-------------|-----------|
| 5.1 打刻機能 | PCブラウザから打刻 | F-01 | UAT-01 | ✅ |
| 5.1 打刻機能 | スマートフォンから打刻 | F-01 | UAT-02 | ✅ |
| ... | ... | ... | ... | ... |

**全ての RFP 要求に対してカバレッジが ✅ であることを確認すること。**

##### 2. テストケース一覧

| テストケースID | テスト名 | 前提条件 | テスト手順 | 期待結果 | 対応要件 |
|-------------|---------|---------|-----------|---------|---------|

##### 3. 主要テストケースの詳細手順

以下のテストケースは画面操作レベルで詳細な手順を記述:
- 出勤打刻（PC / スマートフォン）
- 退勤打刻
- 有給休暇申請・承認
- 月次勤怠サマリーの確認
- CSV 出力
- 36協定アラートの確認

##### 4. 業務シナリオテスト（3本以上）

エンドツーエンドの業務シナリオ:

**シナリオA: 一般従業員の1ヶ月の勤怠管理**
1. 毎日出退勤を打刻（リモートワーク日は位置情報付き）
2. 月中に1日有給を申請・取得
3. 月末に勤怠サマリーを確認
4. 残業時間が36協定の閾値に近づいた場合のアラート確認

**シナリオB: 管理職の部門管理**
1. 部下の休暇申請を承認
2. 部門の勤怠状況をダッシュボードで確認

**シナリオC: 総務部の月次処理**
1. 全社の月次集計を実行
2. 給与計算ソフト向け CSV を出力
3. 有給5日取得義務の未達者を確認

##### 5. 受け入れ基準

- 全テストケースの合格率: 100%
- Critical バグ: 0件
- Major バグ: 3件以下（ワークアラウンドあり）

### Phase 2: セルフレビューと修正

`acceptance_test_plan.md` を読み直し、以下の確認項目を1つずつチェックしてください。
問題が見つかった場合は修正して `acceptance_test_plan.md` を更新し、再度チェックしてください。
**全ての項目が ✅ になるまでこのサイクルを繰り返してください。**

確認項目:
- [ ] RFP セクション5の全要求がトレーサビリティマトリクスに含まれているか
- [ ] 全ての要求のカバレッジが ✅ になっているか（❌ が残っていないか）
- [ ] 主要テストケースの手順が具体的か（画面操作レベルで記述）
- [ ] テスト手順が「動作を確認する」のような曖昧な記述になっていないか
- [ ] 業務シナリオテストが 3 本以上あるか
- [ ] 業務シナリオが現実的な業務フローになっているか
- [ ] 受け入れ基準が定量的に定義されているか

### 振り返り: AI 駆動の自律開発フロー

```
要件分析   ← AI が RFP の曖昧さを自律的に検出・修正
  │
  要件定義   ← AI が機能要件を構造化、セルフレビューで品質保証
    │
    基本設計   ← AI が ER図・API 設計を自律的にレビュー・補完
      │
      詳細設計   ← AI が計算ロジックを自己検算、不整合を自動修正
        │
        コーディング ← AI が TDD で全層実装、テスト失敗を自律デバッグ
        │
        コードレビュー ← AI が 7 観点でレビュー、Critical を自動修正・再テスト
      │
      単体テスト   ← AI が追加テストを生成、全 PASS まで自律修正
    │
    結合テスト   ← AI が API シナリオテストを追加、自律デバッグ
  │
  システムテスト ← AI が非機能要件カバレッジを自律チェック
│
受け入れテスト ← AI が RFP 全要求のカバレッジ 100% を自律保証
```

### このハンズオンで体験したこと

1. **AI は自律的に開発サイクルを回せる** -- 生成・レビュー・修正を1つのプロンプトで実行
2. **人間は「判断」に集中する** -- AI の完了報告を確認し、必要に応じて追加指示を出す
3. **前工程の成果物が次の工程の入力になる** -- `@` でファイルを参照し、一貫性を保った
4. **V字の左右が対応する** -- 設計工程で決めたことが、対応するテストで検証された
5. **セルフレビューが品質を担保する** -- AI 自身がチェックリストで検証・修正するサイクルが重要

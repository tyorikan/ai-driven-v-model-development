```
  >>> 要件分析 <<< ────────────────────── >>> 受け入れテスト <<<

    要件定義 ────────────────────────── システムテスト
```

````
## 作業手順

1. RFP のセクション5「希望する機能」の全項目を抽出する
2. 各機能に対応する要件ID（F-01〜）を紐付ける
3. 各要件に対するテストケースを設計する
4. トレーサビリティマトリクスで全要求のカバレッジを確認する

## 出力形式

### 1. トレーサビリティマトリクス

RFP の要求 → 要件ID → テストケースID の対応表:

| RFP セクション | RFP の要求内容 | 要件ID | テストケースID | カバレッジ |
|--------------|--------------|--------|-------------|-----------|
| 5.1 打刻機能 | PCブラウザから打刻 | F-01 | UAT-01 | ✅ |
| 5.1 打刻機能 | スマートフォンから打刻 | F-01 | UAT-02 | ✅ |
| ... | ... | ... | ... | ... |

**全ての RFP 要求に対してカバレッジが ✅ であることを確認すること。**

### 2. テストケース一覧

| テストケースID | テスト名 | 前提条件 | テスト手順 | 期待結果 | 対応要件 |
|-------------|---------|---------|-----------|---------|---------|

### 3. 主要テストケースの詳細手順

以下のテストケースは画面操作レベルで詳細な手順を記述:
- 出勤打刻（PC / スマートフォン）
- 退勤打刻
- 有給休暇申請・承認
- 月次勤怠サマリーの確認
- CSV 出力
- 36協定アラートの確認

### 4. 業務シナリオテスト（3本以上）

エンドツーエンドの業務シナリオ:

**シナリオA: 一般従業員の1ヶ月の勤怠管理**
1. 毎日出退勤を打刻（リモートワーク日は位置情報付き）
2. 月中に1日有給を申請・取得
3. 月末に勤怠サマリーを確認
4. 残業時間が36協定の閾値に近づいた場合のアラート確認

**シナリオB: 管理職の部門管理**
1. 部下の休暇申請を承認
2. 部門の勤怠状況をダッシュボードで確認

**シナリオC: 総務部の月次処理**
1. 全社の月次集計を実行
2. 給与計算ソフト向け CSV を出力
3. 有給5日取得義務の未達者を確認

### 5. 受け入れ基準

- 全テストケースの合格率: 100%
- Critical バグ: 0件
- Major バグ: 3件以下（ワークアラウンドあり）

## やってはいけないこと

- RFP の要求でカバレッジが ❌ のものを残さないこと（全て ✅ にする）
- テスト手順が「動作を確認する」のような曖昧な記述にならないこと
- 業務シナリオテストを省略しないこと
````

### 振り返り: V字モデルの左右の対応

```
要件分析   ← AI が RFP の曖昧さを 12件以上検出
  │
  要件定義   ← AI が機能要件を構造化、労基法のルールを明文化
    │
    基本設計   ← AI が ER図、API 15本以上、RBAC を設計
      │
      詳細設計   ← AI が計算ロジック、シーケンス図、エラー設計を定義
        │
        コーディング ← AI が TDD でテスト13件 → 実装コード 6ファイルを生成
        │
        コードレビュー ← AI が 7 観点でレビュー、Critical/Major を検出
      │
      単体テスト   ← AI が追加テスト6件を生成（合計19件）
    │
    結合テスト   ← AI が API シナリオテスト 6件を生成
  │
  システムテスト ← AI が性能・セキュリティ・可用性のテスト計画を策定
│
受け入れテスト ← AI が RFP 全要求のカバレッジ 100% を保証
```

### このハンズオンで体験したこと

1. **AI は「たたき台」を高速に生成する** -- 各工程で数分で成果物が生成された
2. **人間は「判断」に集中する** -- AI の出力をレビュー・修正することに時間を使った
3. **前工程の成果物が次の工程の入力になる** -- `@` でファイルを参照し、一貫性を保った
4. **V字の左右が対応する** -- 設計工程で決めたことが、対応するテストで検証された
5. **プロンプト設計が品質を決める** -- 具体的な指示、禁止事項、チェックリストが重要

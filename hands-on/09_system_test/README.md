```
  要件分析 ──────────────────────────────── 受け入れテスト
```

````
## 作業手順

1. 要件定義書の非機能要件（NF-01〜）を全て読む
2. 各非機能要件に対応するテスト項目を設計する
3. テストの実施手順、合格基準、使用ツールを定義する

## 出力形式

### 2. 性能テスト計画

#### 2.1 負荷テスト

| # | テスト項目 | シナリオ | 合格基準 | ツール |
|---|-----------|---------|---------|--------|

具体的な負荷テストシナリオ:

1. **朝の出勤ラッシュ時の打刻負荷**
   - シナリオ: 300名が8:50〜9:10の20分間に一斉に打刻
   - 合格基準: 95パーセンタイルのレスポンスタイム < 2秒
   - ツール: Locust または k6

2. **月次集計バッチの処理性能**
   - シナリオ: 350名 × 1ヶ月分（約7,000件の打刻データ）を一括集計
   - 合格基準: 5分以内に完了

3. **CSV出力の処理性能**
   - シナリオ: 350名 × 12ヶ月分の勤怠データを CSV 出力
   - 合格基準: 30秒以内に完了

### 3. セキュリティテスト計画

| # | テスト項目 | テスト手順 | 合格基準 | ツール |
|---|-----------|-----------|---------|--------|

具体的なテスト項目:
1. JWT認証テスト（トークン期限切れ、改ざん → 401）
2. 認可横断テスト（他者のデータアクセス → 403）
3. SQLインジェクションテスト
4. 個人情報保護テスト（位置情報がログに平文で出力されていないか）
5. OWASP Top 10 チェック（OWASP ZAP でスキャン）

### 4. 可用性テスト計画

1. 稼働時間（打刻は24時間、管理機能は7:00〜23:00）
2. バックアップ/リストア手順と所要時間
3. 障害復旧（RTO の検証）

### 5. データ保持テスト

5年分のデータ（350名 × 5年）でのクエリ性能テスト。

### 6. テストスケジュール

各テストの実施順序と所要日数。

## やってはいけないこと

- テスト項目に合格基準がないものを書かないこと（必ず数値で定義）
- セキュリティテストを省略しないこと
- 性能テストのシナリオが非現実的にならないこと（350名規模を考慮する）
````

## この工程で学ぶこと

| 学習ポイント | 説明 |
|------------|------|
| V字の対応関係 | 要件定義の非機能要件 → システムテストで検証、という対応が明確 |
| 合格基準の数値化 | 「速いこと」ではなく「95パーセンタイル < 2秒」と定量的に定義する |
| 現実的なシナリオ設計 | 350名規模に合った負荷シナリオを設計する（過大な想定は不要） |
| セキュリティテストの網羅 | OWASP Top 10 を基にした体系的なセキュリティテスト設計 |

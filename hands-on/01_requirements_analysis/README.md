# Step 01: 要件分析

## この工程の位置づけ

```
>>> 要件分析 <<< ──────────────────────────── 受け入れテスト
      │                                          │
      要件定義 ────────────────────── システムテスト
```

V字モデルの最上流工程です。RFP（提案依頼書）を受け取り、**曖昧な箇所・不足情報・リスク**を洗い出します。この工程の品質が、最下流の「受け入れテスト」の品質を決定します。

---

## やること

RFP（`rfp.md`）を Gemini CLI に読み込ませ、以下を分析させます:

1. 曖昧な記述の洗い出し
2. 不足情報の特定
3. リスクの評価
4. 顧客への確認事項リストの作成

---

## 手順

### 1. 作業ディレクトリに移動し、Gemini CLI を対話モードで起動

```bash
cd hands-on/01_requirements_analysis
gemini
```

### 2. 以下のプロンプトを入力

> **操作方法**: 下記のプロンプト全体をコピーして、Gemini CLI の対話モードに貼り付けてください。
> 末尾の `@rfp.md` により、同じディレクトリの RFP ファイルが自動的に読み込まれます。

````
あなたは人事・労務管理システムの開発経験が 10年以上あるシニア SE です。
労働基準法、36協定、フレックスタイム制度に精通しています。

以下の RFP（提案依頼書）を分析し、提案・見積もりを行う前に
顧客に確認すべき事項を洗い出してください。

## 作業手順（この順番で分析してください）

1. まず RFP 全体を通読し、システムの目的と対象範囲を把握する
2. 次に、以下の 5 つの分析観点それぞれについて、RFP の各セクションを
   1つずつ照らし合わせながら確認事項を洗い出す
3. 各確認事項に対して、重要度（高/中/低）を判定する
   - 高: これが不明確だと設計着手できない、または見積もりが大きくブレる
   - 中: 基本設計フェーズまでに確定すれば良い
   - 低: 詳細設計以降で確定しても影響が小さい
4. 最後に、確認事項をリスク評価サマリーと優先付き質問リストに整理する

## 分析の観点（5つ全てを必ず実施すること）

1. **曖昧な記述**: 実装方法が複数考えられる箇所
   - 例:「適切なアクセス制御」→ ロールベース？属性ベース？IP制限？
2. **不足情報**: 設計・見積もりに必要だが記載されていない情報
   - 例: フレックスタイム制の清算期間（1ヶ月？3ヶ月？）
3. **リスク**: 技術的リスクやスケジュールリスク
   - 例: 給与大臣との CSV 連携仕様が未確定
4. **前提条件**: 明示されていないが確認が必要な前提
   - 例: スマートフォン打刻時のオフライン対応要否
5. **非機能要件の不足**: 性能・セキュリティの具体的数値
   - 例: 同時打刻数（朝の出勤ラッシュ時）、レスポンス要件

## 出力形式

以下の 3 つのセクションを必ず全て出力してください。

### セクション1: 確認事項一覧（表形式）

| # | カテゴリ | 確認事項 | 重要度 | RFPの該当箇所 | 確認が必要な理由 |
|---|---------|---------|--------|--------------|----------------|

最低 12 件以上の確認事項を洗い出すこと。

### セクション2: リスク評価サマリー

高リスク項目について、「影響」と「対策案」をそれぞれ記述する。

### セクション3: 顧客への質問リスト（優先度順）

「最優先（設計着手前に回答必須）」と「優先（基本設計までに回答必要）」の
2段階に分けて番号付きリストで出力する。

## やってはいけないこと

- RFP に書かれていないことを勝手に仮定して「問題なし」と判断しないこと
- 確認事項を 5 件以下で終わらせないこと（最低 12 件）
- 重要度を全て「中」にしないこと（高/中/低のバランスを取る）
- 労働基準法関連の確認事項を省略しないこと

## 品質チェックリスト（出力前に自己確認すること）

- [ ] 5 つの分析観点すべてから確認事項を出しているか
- [ ] 各確認事項に RFP の具体的な該当箇所を明記しているか
- [ ] 労働基準法・36協定に関する確認事項が含まれているか
- [ ] フレックスタイム制特有の確認事項が含まれているか
- [ ] 顧客質問リストが優先度順に整理されているか

@rfp.md
````

### 3. 生成結果をレビュー

以下の観点でレビューしてください:

- [ ] 残業計算のルール（法定内残業と法定外残業の区別）について確認事項があるか
- [ ] フレックスタイム制の清算期間やコアタイムの扱いについて触れているか
- [ ] 36協定の上限（月45時間・年360時間）の具体的な運用について確認しているか
- [ ] 位置情報の取得に関するプライバシーの懸念を指摘しているか
- [ ] 給与大臣との CSV 連携仕様に関するリスクを指摘しているか

### 4. 結果を保存

対話モードで出力された結果を `ai_analysis.md` というファイル名で保存してください。

```bash
# 方法1: 出力をコピーしてエディタで保存
# 方法2: ワンショット実行でファイルに直接保存する場合
cat rfp.md | gemini -p "（上記と同じプロンプト）" -o text > ai_analysis.md
```

> **重要**: この `ai_analysis.md` は次の Step 02 では直接参照しませんが、
> レビュー時に発見した曖昧な箇所を理解しておくことが、以降の工程の品質に影響します。

## 期待される出力の例

正常に実行できた場合、以下のような出力が得られます:

- **セクション1**: 12件以上の確認事項が表形式で出力される
- **セクション2**: 「給与大臣連携リスク」「労基法解釈リスク」「ピーク負荷リスク」などが記述される
- **セクション3**: 「清算期間」「残業計算ロジック」「CSV仕様」などが最優先に分類される

---

## この工程で学ぶこと

| 学習ポイント | 説明 |
|------------|------|
| AI は網羅的な分析が得意 | 5つの観点を体系的にチェックし、人間の見落としを防ぐ |
| ドメイン知識をプロンプトに含める | 「労働基準法」「36協定」などの専門用語を指定することで、AI の出力品質が向上する |
| 出力形式を明確に指定する | 表形式・リスト形式を指定することで、レビューしやすい出力が得られる |
| 「やってはいけないこと」が重要 | AI がやりがちな失敗（曖昧な分析、件数不足）を明示的に禁止する |

# Step 02: 要件定義

## この工程の位置づけ

```
  要件分析 ──────────────────────────────── 受け入れテスト
    │                                          │
    >>> 要件定義 <<< ──────────────────── システムテスト
```

要件分析で洗い出した確認事項を踏まえ、**機能要件・非機能要件を構造化して定義**します。
ここで定義した要件が、右側の「システムテスト」で検証されます。

---

## やること

RFP を基に、AI が自律的に**要件定義書**を生成・保存・レビュー・修正します。

cd hands-on/02_requirements_definition

あなたは人事・労務管理システムの要件定義に精通したシニア SE です。

以下の RFP を基に、要件定義書を作成してください。

### Phase 1: 成果物の生成と保存

以下の作業手順と出力形式に従って要件定義書を作成し、`requirements.md` に保存してください。

#### 作業手順

1. RFP の機能要件を読み、機能要件 ID（F-01〜）を付与して構造化する
2. 非機能要件を読み、非機能要件 ID（NF-01〜）を付与して構造化する
3. 用語定義を整理する（同じ概念に異なる名前が使われていないか確認）
4. 各要件に受け入れ基準（Acceptance Criteria）を定義する

#### 出力形式

##### 1. 用語定義

システムで使用する用語を以下の形式で定義:

| 用語 | 定義 | 備考 |
|------|------|------|

##### 2. ビジネスルール

労働基準法に基づく計算ルールを明記:

- 法定労働時間: 1日8時間、週40時間
- 残業時間の計算方法（法定内残業と法定外残業の区別）
- 深夜勤務の定義（22:00〜翌5:00）
- 休憩時間の自動控除（6時間超で45分、8時間超で60分 -- 労基法34条）
- 有給休暇の付与日数（勤続年数別: 0.5年=10日, 1.5年=11日, ... 6.5年以上=20日）
- 36協定の上限（月45時間、年360時間）
- フレックスタイム制の所定労働時間計算（清算期間1ヶ月）
- 時間単位有給の年間上限（5日分 = 40時間）
- 打刻の丸め処理（1分単位、丸め処理なし）

##### 3. 機能要件一覧

| 要件ID | カテゴリ | 要件名 | 要件詳細 | 受け入れ基準 | 優先度 |
|--------|---------|--------|---------|-------------|--------|

要件詳細は曖昧な表現を避け、具体的な数値や条件を含めること。

**良い例**: 「1日の実労働時間が8時間を超えた部分を法定外残業として計算し、
22:00〜翌5:00の勤務時間を深夜勤務として別途集計する」

**悪い例**: 「残業時間を適切に計算する」

##### 4. 非機能要件一覧

| 要件ID | カテゴリ | 要件名 | 要件詳細 | 測定基準 |
|--------|---------|--------|---------|---------|

##### 5. 画面一覧

| 画面ID | 画面名 | 利用者 | 主な機能 |
|--------|--------|--------|---------|

### Phase 2: セルフレビューと修正

`requirements.md` を読み直し、以下の確認項目を1つずつチェックしてください。
問題が見つかった場合は修正して `requirements.md` を更新し、再度チェックしてください。
**全ての項目が ✅ になるまでこのサイクルを繰り返してください。**

確認項目:
- [ ] 全ての機能要件に一意のIDが付与されているか
- [ ] 要件詳細に曖昧な表現（「適切に」「必要に応じて」「等」）がないか
- [ ] 受け入れ基準が具体的で検証可能か（テスト可能か）
- [ ] 残業計算のビジネスルール（法定内/法定外の区別）が明記されているか
- [ ] 有給休暇の付与日数表が勤続年数別に記載されているか
- [ ] フレックスタイム制の計算ルールが具体的に記載されているか
- [ ] 36協定の上限（月45時間・年360時間）が明記されているか
- [ ] 非機能要件に具体的な数値目標が含まれているか
- [ ] 全ての機能要件に受け入れ基準が記載されているか

## やってはいけないこと

- 要件に曖昧な表現（「適切に」「必要に応じて」「等」）を使わないこと
- 受け入れ基準のない機能要件を書かないこと
- 非機能要件に具体的な数値目標がないものを書かないこと
- 労働基準法の計算ルールを省略しないこと

## この工程で学ぶこと

| 学習ポイント | 説明 |
|------------|------|
| 曖昧な要件を具体化する | 「適切に」を禁止し、具体的な数値・条件を定義させる |
| Good/Bad の例示が効果的 | 良い例と悪い例を並べることで、AI の出力品質が大幅に向上する |
| AI の自律的な品質保証 | セルフレビューで曖昧表現を自動検出・修正させる |
| 前工程の成果物を入力にする | RFP を `@` で参照し、一貫性のある要件定義を生成する |

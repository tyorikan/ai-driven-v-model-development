# 09 システムテスト: AI プロンプト

## この工程の目的

要件定義書の**非機能要件（性能・可用性・セキュリティ）が満たされているか**を検証する工程です。
AI は要件定義書の非機能要件から、テスト計画とチェックリストを自動生成します。

## V字モデルにおける位置づけ

```
要件定義 ←──── 対応 ────→ システムテスト
  │                           │
  非機能要件                   性能テスト
  (NF-01〜NF-10)             セキュリティテスト
                              可用性テスト
```

**システムテストは要件定義書の非機能要件（NF-01〜NF-10）を検証基準とします。**

## AI への指示（プロンプト）

```
あなたはシステムテストのスペシャリストです。Web API の性能テスト（Locust）、
セキュリティテスト（OWASP ZAP）、可用性テスト（フェイルオーバー検証）の
実施経験が豊富です。

@requirements.md の非機能要件（セクション 4、要件ID: NF-01〜NF-10）を基に、
システムテスト計画書を作成してください。

## 作業手順（この順番で計画を作成すること）

1. 非機能要件 NF-01〜NF-10 を全て読み、テストカテゴリ（性能/セキュリティ/可用性）
   に分類する
2. 各カテゴリについてテスト項目を設計する
3. 各テスト項目に対して、テスト方法・合格基準・使用ツールを明記する
4. テスト結果を記録するテンプレートを作成する

## 出力するシステムテスト計画書の構成

### 1. 性能テスト計画

以下の表形式で記述:
| テストID | 対象API | 目標値 | 対応要件 | テスト条件 |

必ず以下の 4 つのテストを含めること:
- PT-01: 検査結果一覧（GET /inspections）の応答速度
  - 目標値: 2秒以内（NF-01）
  - テスト条件: inspection_result テーブルに 10,000 件格納時
- PT-02: 検査結果登録（POST /inspections）の応答速度
  - 目標値: 1秒以内（NF-02）
  - テスト条件: 検査明細 10 項目の検査登録
- PT-03: ダッシュボード描画の応答速度
  - 目標値: 3秒以内（NF-03）
  - テスト条件: 30日分のデータ集計
- PT-04: 同時接続テスト
  - 目標値: 70ユーザー同時接続で正常応答（NF-04）

負荷テストスクリプトを Locust の Python コードとして記述すること。
具体的な API パスとリクエストボディを含めること。

合格基準を以下の表形式で記述:
| 指標 | 合格基準 |
- 平均応答時間、95パーセンタイル、エラー率、スループットの4指標

### 2. セキュリティテスト計画

以下の表形式で記述:
| テストID | カテゴリ | テスト内容 | 対応要件 | 合格基準 |

必ず以下の 6 つのテストを含めること:
- ST-01: 無効なトークンでの API アクセス → 401 返却（NF-07）
- ST-02: 権限不足のロールでのマスタ管理 API アクセス → 403 返却（NF-07）
- ST-03: SQL インジェクション文字列の入力 → 422 返却
- ST-04: 検査結果の改ざん防止（変更履歴の保持確認）→ revision 増加（NF-09）
- ST-05: 全 API 操作の監査ログ記録確認（NF-08）
- ST-06: XSS（スクリプトタグを含む不良内容の登録）→ エスケープ確認

ST-04 については、具体的なテスト手順（5ステップ）を記述すること。

### 3. 可用性テスト計画

以下の表形式で記述:
| テストID | 障害シナリオ | テスト方法 | 期待結果 | 対応要件 |

必ず以下の 4 つのテストを含めること:
- AT-01: DB フェイルオーバー → 30秒以内に復旧（NF-06）
- AT-02: API サーバー停止 → 自動復旧（NF-06）
- AT-03: MES 連携障害 → リトライ後にアラート通知（F-08）
- AT-04: 通常の 3 倍の負荷 → オートスケール（NF-04）

稼働率の計測方法を以下の計算式で記述:
稼働率 = (計測期間の総分数 - ダウンタイム分数) / 計測期間の総分数 × 100

### 4. テスト結果記録テンプレート

以下の表形式:
| テストID | テスト名 | 実行日 | 実行者 | 結果 | 測定値 | 合格基準 | 判定 | 備考 |

全テスト項目（PT-01〜04, ST-01〜06, AT-01〜04）の行を含めること。

## やってはいけないこと

- 非機能要件 NF-01〜NF-10 に対応しないテスト項目を書かないこと
- 合格基準を「正常に動作すること」のような曖昧な記述にしないこと
  （具体的な数値・条件を明記する）
- 負荷テストの Locust スクリプトで実際の API パスと異なるパスを使わないこと
- テスト結果記録テンプレートのテスト項目を省略しないこと（全件記載）

## 品質チェックリスト（出力前に自己確認すること）

- [ ] NF-01〜NF-10 の全てに対応するテスト項目があるか
- [ ] 全テスト項目に具体的な合格基準（数値）があるか
- [ ] Locust スクリプトが実際の API 設計と整合しているか
- [ ] テスト結果記録テンプレートに全テスト項目の行があるか
- [ ] セキュリティテスト ST-04 の手順が 5 ステップで記述されているか
```

## AI 活用のポイント

| 観点 | 従来の方法 | AI 活用後 | 改善効果 |
|------|-----------|----------|---------|
| 所要時間 | テスト計画の作成に1〜2週間 | AIが要件定義から数分でテスト計画を生成 | **90%以上の時間削減** |
| 性能テスト | シナリオ作成は経験に依存 | 業界標準のテストパターンを提案 | 属人性排除 |
| セキュリティ | テスト観点が属人的 | OWASP Top 10を体系的にカバー | 網羅性向上 |

## 人間の役割

1. **テスト環境の構築**: 本番に近いインフラで性能テストを実施
2. **合格基準の妥当性判断**: 目標値が業務要件として妥当か確認
3. **テスト結果の分析**: ボトルネックの特定と改善策の立案

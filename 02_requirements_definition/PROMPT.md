# 02 要件定義: AI プロンプト

## この工程の目的

RFP 分析で洗い出した確認事項への回答を踏まえ、**開発可能な粒度の要件定義書**を作成する工程です。
AI は RFP と顧客回答から、構造化された要件定義書のたたき台を自動生成します。

## AI への指示（プロンプト）

```
あなたは製造業（自動車部品製造業）向けシステム開発のシニアSEです。
IATF 16949 認証工場の品質管理業務に精通しています。

以下の RFP と顧客からの回答を基に、要件定義書を作成してください。

## 入力情報

- @rfp.md : 元の RFP（製造ライン品質管理システム）
- @customer_answers.md : RFP 分析後の顧客回答

## 作業手順（この順番で作成してください）

1. まず RFP と顧客回答を突合し、全ての要求事項を漏れなく洗い出す
2. 要求事項を機能要件（F-XX）と非機能要件（NF-XX）に分類する
3. 各機能要件について、入力・出力・処理ルールを具体的に定義する
4. 業務フローを Mermaid で作図し、機能間の関連を明確にする
5. 用語定義表を作成し、日本語と英語（システム内部名称）を対応させる
6. 最後に、要件の整合性を自己チェックする

## 出力する要件定義書の構成

### 1. システム概要
- 目的（RFP の「背景」と「目的」を基に、1〜2文で記述）
- 対象範囲（対象ライン、対象工程、データ量を表形式で記述）
- 利用者（ロール、人数、主な操作を表形式で記述）

### 2. 業務フロー
- 検査業務フロー: 製造ロット完成 → 検査値入力 → 合否判定 → 記録保存
  を Mermaid flowchart TD で記述すること
- 出荷判定フロー: 出荷検査完了 → 品質サマリー → 出荷可否判定
  を Mermaid flowchart TD で記述すること

### 3. 機能要件（F-XX 形式で番号付与）
各機能について以下の表形式で記述すること:
| 項目 | 内容 |
|------|------|
| 概要 | 機能の説明（1〜2文） |
| 入力 | 入力データ項目をカンマ区切りで列挙 |
| 出力 | 出力データ項目をカンマ区切りで列挙 |
| 処理ルール | 具体的な業務ルール（数値・条件を明記） |

必須の機能要件:
- F-01: 検査基準マスタ管理（品目ごとの上限値・下限値）
- F-02: 検査結果入力（タブレット対応、合否自動判定）
- F-03: 不良記録管理（不良区分・処置の記録）
- F-04: 品質ダッシュボード（不良率グラフ、更新頻度を明記）
- F-05: 品質アラート通知（閾値・通知方法・通知先を明記）
- F-06: ロットトレーサビリティ（全工程の検査履歴）
- F-07: 品質レポート出力（CSV/Excel）
- F-08: MES 連携（FactoryTalk との REST API 連携）

### 4. 非機能要件
以下の表形式で記述すること:
| カテゴリ | 要件ID | 要件 | 目標値 |
必ず性能・可用性・セキュリティ・データ保持の4カテゴリを含めること。

### 5. 外部連携要件
MES との連携を Mermaid sequenceDiagram で記述すること。

### 6. 用語定義
| 日本語 | 英語（システム内統一名称） | 説明 |
の表形式で全業務用語を定義すること。

## 具体的な記述例

### 処理ルールの良い例と悪い例

悪い例: 「測定値が基準範囲外であれば不合格とする」
良い例: 「測定値が下限値以上かつ上限値以下であれば合格。
         1項目でも不合格があれば、そのロットの検査結果は不合格とする」

悪い例: 「不良率が高い場合にアラートを出す」
良い例: 「直近1時間の不良率が設定閾値（デフォルト: 3%）を超えた場合に
         アラートを発行する。通知先は該当ラインの品質管理者。
         通知方法はシステム内通知 + メール」

## やってはいけないこと

- 「適切に」「必要に応じて」「等」などの曖昧な表現を使わないこと
- 数値目標のない非機能要件を書かないこと（例: ×「高速に表示」 → ○「2秒以内に表示」）
- 機能要件に処理ルールを書かずに概要だけで済ませないこと
- RFP に記載された要求を 1 つでも漏らさないこと
- 顧客回答の内容を無視しないこと

## 品質チェックリスト（出力前に自己確認すること）

- [ ] RFP の「希望する機能」6 項目が全て機能要件（F-XX）にマッピングされているか
- [ ] 顧客回答の内容（検査項目数、ロット番号体系等）が要件に反映されているか
- [ ] 全ての処理ルールに具体的な数値・条件が含まれているか
- [ ] 非機能要件に性能・可用性・セキュリティ・データ保持の 4 カテゴリがあるか
- [ ] 用語定義の英語名が snake_case で統一されているか
- [ ] 業務フローが Mermaid で正しく記述されているか
```

## 顧客回答の例（customer_answers.md として想定）

```markdown
# 顧客回答

## Q1: 検査項目数と検査基準
- ブレーキパッド: 寸法検査5項目、硬度検査1項目、外観検査3項目（計9項目）
- ディスクローター: 寸法検査8項目、表面粗さ1項目、外観検査2項目（計11項目）
- 検査基準値は品目マスタで管理。上限値・下限値で合否判定

## Q2: MES
- 製品名: FactoryTalk (Rockwell Automation)
- REST API あり。ロット番号・製造実績データを取得可能
- API仕様書は別途提供

## Q3: IATF 16949 の電子記録要件
- 電子署名は不要（検査員IDの記録で可）
- 改ざん防止（記録の変更履歴）は必須
- 記録の保持期間: 10年

## Q4: 1日あたりの検査件数
- ブレーキパッド: 約500件/日
- ディスクローター: 約300件/日
- 組立ライン: 約200件/日
- 合計: 約1,000件/日

## Q5: ロット番号体系
- 形式: YYYYMMDD-LINE-SEQ（例: 20260401-BP-001）
- MES で自動採番される
```

## AI 活用のポイント

| 観点 | 従来の方法 | AI 活用後 | 改善効果 |
|------|-----------|----------|---------|
| 所要時間 | SEが1〜2週間かけて要件を整理・文書化 | AIが構造化された要件定義書を数分で生成 | **90%以上の時間削減** |
| 網羅性 | 要件の抜け漏れは人間のレビューに依存 | AIが RFP の全要求を体系的にマッピング | 抜け漏れ防止 |
| 一貫性 | 用語の不統一が後工程で問題化 | AIが用語定義表を自動整理 | 手戻り削減 |

## 人間の役割

1. **顧客との合意**: AI が生成した要件定義書を顧客と確認し、認識齟齬をなくす
2. **優先度の判断**: 要件の優先度（Must/Should/Could）は人間が業務観点で判断
3. **実現可能性の検証**: 技術的な実現可能性は人間が判断する

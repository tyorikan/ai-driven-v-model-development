# 10 受け入れテスト: AI プロンプト

## この工程の目的

RFP の要求事項が**全て実現されているか**を、業務シナリオに基づいて検証する工程です。
AI は RFP と要件定義書から、業務ユーザーが実行する受け入れテストのシナリオを自動生成し、
RFP の各要求とテストケースのトレーサビリティマトリクスを作成します。

## V字モデルにおける位置づけ

```
要件分析（RFP） ←──── 対応 ────→ 受け入れテスト
  │                                  │
  顧客の要求事項                      業務シナリオでの検証
  業務課題の解決                      課題が解決されたかの確認
```

**受け入れテストは RFP の要求事項を検証基準とします。
RFP に記載された全ての要求が実現されていることを確認します。**

## AI への指示（プロンプト）

```
あなたは製造業向けシステムの品質保証（QA）スペシャリストです。
IATF 16949 認証工場での UAT（User Acceptance Test）の実施経験が豊富です。

@rfp.md（RFP）と @requirements.md（要件定義書）を基に、
受け入れテスト（UAT）の仕様書を作成してください。

## 作業手順（この順番で作成すること）

1. RFP の「希望する機能」（6項目）と「技術要件」（3項目）を全て抽出する
2. 各要求に対応する機能要件（F-XX）を特定し、トレーサビリティマトリクスを作成する
3. 各機能要件に対して、業務ユーザーが実行するテストシナリオを設計する
4. RFP の「背景」に記載された 4 つの課題が解決されたことを確認するチェックリストを作成する
5. 受け入れ判定基準を定義する

## 出力する受け入れテスト仕様書の構成

### 1. トレーサビリティマトリクス

以下の表形式で、RFP の要求と機能要件とテストケースの対応を記述:
| RFP 要求 | 対応する機能要件 | 受け入れテストケース | カバレッジ |

RFP の以下の要求を全てマッピングすること（1つも漏らさないこと）:
- 検査結果をタブレットから入力できること → F-02 → UAT-01, UAT-02
- 検査結果の合否を自動判定できること → F-02 → UAT-01, UAT-03
- 不良率をリアルタイムに表示するダッシュボードがあること → F-04 → UAT-05
- 品質アラート通知 → F-05 → UAT-06
- 検査データを CSV でエクスポートできること → F-07 → UAT-07
- ロット単位でのトレーサビリティが確保できること → F-06 → UAT-08
- MES との連携 → F-08 → UAT-09
- データは 10 年間保存すること → NF-10 → ST-04

最終行にカバレッジ（100% であること）を明記する。

### 2. 業務シナリオテスト

以下の 4 つのシナリオを必ず含めること。
各シナリオは以下の形式で記述:

#### UAT-XX: [シナリオ名]
| 項目 | 内容 |
|------|------|
| 実施者 | [ロール名] |
| 前提条件 | [具体的な前提] |
| 想定所要時間 | [分] |

操作手順（表形式）:
| # | 操作 | 期待結果 |

確認ポイント（チェックリスト形式）:
- [ ] [確認事項]

必須シナリオ:
- UAT-01: 検査員による工程内検査（正常系）
  - ロット選択 → 検査値入力 → 合否自動判定 → 結果保存
  - 全 7 ステップの操作手順を記述
- UAT-02: 検査員による工程内検査（不合格ケース）
  - 基準範囲外の値を入力 → 不合格表示 → 不良記録入力 → 保存
  - 全 7 ステップの操作手順を記述
- UAT-05: 品質管理者によるダッシュボード確認
  - 不良率グラフ確認 → 期間切替 → パレート図 → アラート一覧
  - 全 5 ステップの操作手順を記述
- UAT-08: ロットトレーサビリティの確認
  - ロット番号検索 → 全工程の検査履歴確認 → CSV 出力
  - 全 4 ステップの操作手順を記述

### 3. RFP 課題解決の確認

RFP の「背景」セクションに記載された以下の 4 つの課題について、
解決されたことを定量的に確認するチェックリストを作成:

| # | RFP記載の課題 | 解決の確認方法 | 合格基準 | 結果 |

4 つの課題:
1. 検査結果の集計に毎日2時間かかる
2. 不良品の傾向分析ができない
3. 顧客からの品質データ提出要求に迅速に対応できない
4. 検査記録の保管が紙ベースで過去データの検索に時間がかかる

### 4. 受け入れ判定基準

合格条件（全条件を満たすこと）を Box 形式で記述:
1. トレーサビリティマトリクスの全項目が「対応済み」
2. 全業務シナリオテストが「合格」
3. RFP 課題解決の全項目が「合格」
4. システムテストが「合格」
5. 業務ユーザーの最終承認

## 制約（重要）

- テストケースは**業務ユーザー（非エンジニア）が実行可能**なレベルで記述すること
  - ×「POST /api/v1/inspections を呼ぶ」
  - ○「検査入力画面で、外径寸法の測定値に 12.25 を入力する」
- 技術用語を避け、業務用語で記述すること
  - ×「HTTP 201 が返る」
  - ○「検査結果が保存され、完了メッセージが表示される」
- 操作手順の各ステップに期待結果を必ず記述すること
- 確認ポイントはチェックリスト形式（ - [ ] ）で記述すること

## やってはいけないこと

- RFP の要求を 1 つでもトレーサビリティマトリクスから漏らさないこと
- 操作手順を「検査を実行する」のように曖昧に書かないこと
  （画面操作レベルの具体的な手順を書く）
- 技術用語（API、JSON、HTTP ステータス等）をテストシナリオに使わないこと
- 期待結果を「正しく動作すること」のように曖昧に書かないこと
- RFP 課題解決の確認を省略しないこと

## 品質チェックリスト（出力前に自己確認すること）

- [ ] RFP の「希望する機能」6 項目が全てトレーサビリティマトリクスに含まれているか
- [ ] カバレッジが 100% と明記されているか
- [ ] 4 つの必須シナリオ（UAT-01, 02, 05, 08）が全て含まれているか
- [ ] 全シナリオの操作手順が画面操作レベルで具体的に記述されているか
- [ ] RFP の 4 つの課題に対する解決確認チェックリストがあるか
- [ ] テストシナリオに技術用語が含まれていないか
- [ ] 受け入れ判定基準が明記されているか

@rfp.md
@requirements.md
```

## AI 活用のポイント

| 観点 | 従来の方法 | AI 活用後 | 改善効果 |
|------|-----------|----------|---------|
| 所要時間 | 業務担当者とSEが1〜2週間かけて作成 | AIが数分で業務シナリオを生成 | **90%以上の時間削減** |
| トレーサビリティ | RFPとの対応関係は手作業で確認 | マトリクスを自動生成 | 漏れ防止 |
| 網羅性 | テスト漏れのリスクが高い | RFP 全要求のカバレッジを自動チェック | 100%カバレッジ保証 |

## 人間の役割

1. **業務観点のレビュー**: AI が生成したシナリオが実際の業務と合致するか確認
2. **テストの実行**: 受け入れテストは必ず**業務ユーザー自身**が実行する
3. **合否判定**: 最終的な受け入れ判定は人間（顧客の品質管理部門長）が行う

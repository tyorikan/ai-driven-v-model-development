# 03 基本設計: AI プロンプト

## この工程の目的

要件定義書を基に、**システムの全体構造（アーキテクチャ、DB設計、API設計）** を決定する工程です。
AI は要件定義書から基本設計書のたたき台を自動生成します。

## AI への指示（プロンプト）

```
あなたは製造業向け Web システムのシニアアーキテクトです。
Google Cloud + Python (FastAPI) + PostgreSQL での設計経験が豊富です。

@requirements.md（要件定義書）を基に、基本設計書を作成してください。

## 作業手順（この順番で設計してください）

1. 要件定義書の機能要件（F-01〜F-08）と非機能要件（NF-01〜NF-10）を
   全て読み、システムに必要なコンポーネントを洗い出す
2. コンポーネント構成図を Mermaid graph TD で作成する
3. 要件定義書の用語定義を基に、テーブル設計（ER 図）を作成する
4. 各機能要件に対応する API エンドポイントを設計する
5. 画面一覧を作成し、各画面が使用する API を紐づける
6. 最後に、要件定義書との整合性を自己チェックする

## 出力する基本設計書の構成

### 1. システムアーキテクチャ

コンポーネント構成図を Mermaid graph TD で記述すること。
以下のコンポーネントを必ず含めること:
- クライアント（タブレット、PC ブラウザ）
- ロードバランサー
- バックエンド API（Cloud Run / FastAPI）
- データベース（Cloud SQL / PostgreSQL 16）
- 外部システム（MES: FactoryTalk、Active Directory、メールサーバー）

技術スタック一覧を以下の表形式で記述:
| レイヤー | 技術 | バージョン | 選定理由 |

### 2. データベース設計

ER 図を Mermaid erDiagram で記述すること。
要件定義書の用語定義に基づき、以下のテーブルを必ず含めること:
- item（品目）
- inspection_standard（検査基準）
- lot（ロット）
- inspection_result（検査結果）
- inspection_detail（検査明細）
- defect_record（不良記録）
- quality_alert（品質アラート）
- shipment_decision（出荷判定）

各テーブルのカラム定義を以下の表形式で記述:
| カラム名 | 型 | 制約 | 説明 |

型は PostgreSQL の型を使用すること:
- 文字列: VARCHAR(n)
- 数値: INTEGER, NUMERIC(p,s)
- 日時: TIMESTAMP
- 真偽値: BOOLEAN
- UUID: UUID
- 長文: TEXT

インデックス設計を以下の表形式で記述:
| テーブル | インデックス名 | カラム | 理由 |

### 3. API 設計

エンドポイント一覧を以下の表形式で記述:
| メソッド | パス | 機能 | 対応要件 |

パスは /api/v1/ で始めること。
パスは kebab-case（例: /api/v1/inspection-results）で統一すること。

主要 API（検査結果登録 POST /api/v1/inspections）のリクエスト/レスポンスを
JSON 形式で具体的に記述すること。ロット番号は "20260401-BP-001" のような
実際のデータ例を使うこと。

エラーレスポンスの形式を以下に統一:
{ "error_code": "STRING", "detail": "説明文", "params": {} }

### 4. 画面一覧

以下の表形式で記述:
| 画面ID | 画面名 | 概要 | 主な使用API | 利用者 |

## 具体的な設計判断の指針

- 検査結果の改ざん防止のため、inspection_result には revision カラム
 （変更版数）を持たせ、物理削除を禁止する設計とすること
- 検査明細（inspection_detail）は検査結果（inspection_result）の子テーブルとし、
  1対多のリレーションとすること
- ロット番号は MES から連携される値をそのまま PK として使用すること
  （UUID を PK にしない）
- ダッシュボードの集計は SQL の GROUP BY で実装する設計とし、
  マテリアライズドビューは Phase 2 以降で検討

## やってはいけないこと

- 要件定義書に記載された用語定義と異なるテーブル名・カラム名を使わないこと
- 外部キー制約を省略しないこと
- API のレスポンス例に "..." や "省略" を使わないこと（具体的な値で書く）
- ER 図で Mermaid の構文エラーが出る記法を使わないこと
- 非機能要件（NF-01〜NF-10）を無視した設計をしないこと

## 品質チェックリスト（出力前に自己確認すること）

- [ ] 機能要件 F-01〜F-08 の全てに対応する API エンドポイントがあるか
- [ ] 用語定義の英語名とテーブル名・カラム名が一致しているか
- [ ] ER 図のリレーション（1対多、1対1）が正しいか
- [ ] 画面一覧の各画面から使用する API が API 設計に存在するか
- [ ] インデックス設計が非機能要件（検索 3 秒以内等）を考慮しているか
```

## AI 活用のポイント

| 観点 | 従来の方法 | AI 活用後 | 改善効果 |
|------|-----------|----------|---------|
| 所要時間 | アーキテクトが1〜2週間かけて設計 | AIが数分で設計書のたたき台を生成 | **90%以上の時間削減** |
| ER図作成 | ER図の作成に半日以上 | Mermaid形式のER図を自動生成 | 大幅な工数削減 |
| 一貫性 | API設計の一貫性は人間のレビューに依存 | 命名規則・パス設計の一貫性を担保 | 手戻り削減 |

## 人間の役割

1. **アーキテクチャの妥当性検証**: 性能・コスト・保守性の観点で AI 設計をレビュー
2. **技術選定の判断**: AI の提案する技術スタックが組織の制約と合致するか確認
3. **設計のトレードオフ判断**: 正規化 vs 非正規化、同期 vs 非同期等の判断は人間が行う

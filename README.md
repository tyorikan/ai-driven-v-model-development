# V字モデル x AI駆動開発 実践例

## 製造ライン品質管理システム（Manufacturing Quality Management System）

本プロジェクトは、**製造工程における品質管理システム**を題材に、V字モデルの全10工程で AI をどのように活用するかを具体的に示した実践例です。

---

## なぜ AI x V字モデルなのか ── 従来開発との決定的な違い

### 従来のV字開発の課題

```
従来の開発者が抱える「あるある」

  要件分析   「RFP を3日かけて読んだが、結局ヒアリングで聞き漏れが発覚…」
  要件定義   「要件定義書を2週間かけて書いたが、曖昧な表現が残り手戻り発生…」
  基本設計   「ER図を半日かけて描いたが、テーブル名の不統一に後から気づいた…」
  詳細設計   「処理フローを書くのに1週間、でも実装時に抜け漏れが見つかる…」
  コーディング「テストは実装後に書こう → 結局時間がなくてテスト不足…」
  コードレビュー「レビュアーの経験によって指摘の質がバラバラ…」
  テスト     「テスト仕様書の作成に2週間 → テスト実施で初めて仕様漏れ発覚…」
```

### AI 活用で何が変わるか

```
┌──────────────────────────────────────────────────────────────────────┐
│                                                                      │
│   AI は「たたき台」を高速に生成する。人間は「判断」に集中する。          │
│                                                                      │
│   従来: 文書を「ゼロから書く」作業に時間の 70% を消費                   │
│   AI後: 文書の「レビュー・修正」に集中。作成時間は 1/10 以下に          │
│                                                                      │
└──────────────────────────────────────────────────────────────────────┘
```

## V字モデルとは

V字モデルは、ソフトウェア開発の各工程を「設計フェーズ」と「テストフェーズ」が対応する形で表現した開発プロセスモデルです。

```
要件分析 ──────────────────────────────────────── 受け入れテスト
  │         RFP の要求を検証                            │
  要件定義 ──────────────────────────────── システムテスト
    │         非機能要件を検証                      │
    基本設計 ──────────────────────────── 結合テスト
      │         API 連携を検証                │
      詳細設計 ──────────────────── 単体テスト
        │         メソッド単位で検証       │
        コーディング ──── コードレビュー
```

**核心**: 左側で「決めたこと」を、右側の対応するテストで「検証」する。
AI はこの対応関係を自動的にトレースし、漏れを防ぎます。

## AI 活用の 5 つの要点

### 要点 1: AI は「ゼロからの文書作成」を自動化する

```
従来: 白紙の状態から要件定義書を書き始める（最も時間がかかる作業）
AI後: AI が 80% 完成した文書を生成 → 人間が残り 20% を修正・補完

効果: 文書作成の工数を 80〜90% 削減
```

### 要点 2: AI は「網羅性チェック」を機械的に担保する

```
従来: 「RFP の要求を全てカバーしているか？」は人間のレビューに依存
AI後: AI がトレーサビリティマトリクスを自動生成し、カバレッジ 100% を保証

効果: 要件の抜け漏れによる手戻りを防止
```

具体例: 受け入れテスト（10_acceptance_test）では、RFP の 8 つの要求に対して
テストケースが漏れなく対応していることをマトリクスで自動確認しています。

### 要点 3: AI は「属人性」を排除する

```
従来: レビューの質はレビュアーの経験に依存。指摘がバラバラ
AI後: 7 つの観点（仕様整合性、セキュリティ、パフォーマンス、エラー処理、
      型安全性、命名規約、アーキテクチャ）で毎回一定品質のレビュー

効果: チームメンバーの経験差に関わらず一定品質を維持
```

### 要点 4: AI は「テスト設計」を自動化し TDD を実現可能にする

```
従来: テストコードは実装後に書く → 時間切れでテスト不足になりがち
AI後: 設計書からテストコードを先に生成（TDD）→ テスト駆動で実装

効果: テストカバレッジの向上、品質の担保
```

具体例: 05_coding では、PROMPT.md の指示通り、テストコード（11ケース）を
先に生成してから実装コードを生成する TDD を実践しています。

### 要点 5: AI は「判断」はしない ── 人間の役割が変わる

```
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│   AI の役割: 文書の生成、網羅性チェック、パターン検出           │
│   人間の役割: 業務判断、顧客合意、リスク評価、最終承認         │
│                                                              │
│   AI は「たたき台を作る職人」                                  │
│   人間は「それを見て判断する意思決定者」                       │
│                                                              │
│   役割が「作業者」から「意思決定者」に変わる。                  │
│   これが AI 駆動開発の本質。                                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## プロンプト設計の要点 ── どんな LLM でも品質を出すために

各工程の PROMPT.md には、AI から高品質な出力を引き出すための 5 つの工夫が施されています。

| # | 工夫 | 説明 | 例 |
|---|------|------|-----|
| 1 | **ステップバイステップ指示** | 「作業手順（この順番で実施）」として番号付きで指示 | 「1. まず RFP 全体を通読し…」 |
| 2 | **具体的な出力形式の指定** | 表形式・Mermaid記法・コード形式を明示 | 「以下の表形式で記述すること:」 |
| 3 | **Good/Bad の具体例** | 良い例と悪い例を並べて品質基準を明示 | 「悪い例: 適切に処理する / 良い例: 下限値以上かつ上限値以下なら合格」 |
| 4 | **禁止事項（やってはいけないこと）** | AI がやりがちなミスを明示的に禁止 | 「曖昧な表現を使わないこと」 |
| 5 | **品質チェックリスト** | 出力前の自己確認項目を明示 | 「□ 機能要件 F-01〜F-08 が全てカバーされているか」 |

## 対象システムの概要

**製造ライン品質管理システム** は、自動車部品工場（IATF 16949 認証）の
製造ラインで発生する品質データを管理するシステムです。

### 業務の流れ

```
原材料受入 → 製造工程 → 品質検査 → 出荷判定
    │            │           │          │
    入荷検査    工程内検査   最終検査    出荷可否判定
    記録        記録         記録        記録
         ↓           ↓          ↓
    ┌────────────────────────────────┐
    │    品質管理システム（本システム）   │
    │                                │
    │  ・検査結果の記録・管理           │
    │  ・不良率の集計・可視化           │
    │  ・品質アラートの通知             │
    │  ・品質レポートの出力             │
    └────────────────────────────────┘
```

### このシステムで解決する課題

| # | 現状の課題 | AI 駆動開発で実現すること |
|---|-----------|----------------------|
| 1 | 検査結果の集計に毎日2時間 | ダッシュボードでリアルタイム可視化 |
| 2 | 不良品の傾向分析ができない | パレート図・推移グラフで自動分析 |
| 3 | 顧客への品質データ提出が遅い | ロット検索 → CSV出力が3分で完了 |
| 4 | 紙ベースで過去データ検索が困難 | 全検査データを10年間デジタル保存 |

## 読み方のガイド

各フォルダには以下の 2 種類のファイルがあります:

| ファイル | 内容 | 読み方 |
|---------|------|--------|
| `PROMPT.md` | AI への指示文（プロンプト）の実例 | **「AI にどう指示するか」** を学ぶ |
| その他のファイル | AI が生成した成果物 | **「AI が何を出力するか」** を確認する |

### おすすめの読み順

1. まずこの README を通読する
2. `01_requirements_analysis/PROMPT.md` → `ai_analysis.md` の順で読む（最も分かりやすい工程）
3. 興味のある工程の `PROMPT.md` → 成果物を読む

### 開発チームでの活用方法

1. **PROMPT.md をコピーして自プロジェクトに適用**: プロンプトのテンプレートとして使用
2. **プロンプトの「作業手順」「禁止事項」「品質チェックリスト」を自プロジェクト向けにカスタマイズ**
3. **AI の出力をレビュー・修正するワークフローを定着させる**

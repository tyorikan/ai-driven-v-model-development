# 06 コードレビュー: AI プロンプト

## この工程の目的

実装されたコードを**仕様書・コーディング規約・セキュリティの観点**でレビューする工程です。
AI は人間のレビュアーに先立ち、機械的にチェック可能な項目を自動で検出します。

## AI への指示（プロンプト）

```
あなたはシニアコードレビュアーです。Python (FastAPI) の Web API 開発に
10 年以上の経験があり、OWASP Top 10、SQLAlchemy のパフォーマンス最適化、
Pydantic v2 のベストプラクティスに精通しています。

以下のコードを、仕様書（@detailed_design.md）とコーディング規約に照らして
レビューしてください。

## 作業手順（この順番でレビューすること）

1. まず @detailed_design.md の処理フローとエラーコード一覧を確認する
2. 各ソースファイルを 1 つずつ読み、以下の 7 観点でチェックする
3. 指摘事項を表形式でまとめる
4. 総合評価（合格/修正後マージ可/差し戻し）を判定する

## レビュー観点（7つ全てを必ずチェックすること）

### 1. 仕様との整合性
- 詳細設計書の処理フロー（番号付きステップ）と実装の処理順序が一致しているか
- エラーコード（LOT_NOT_FOUND, DUPLICATE_INSPECTION 等）が設計通りか
- Pydantic スキーマのフィールド名・バリデーションが設計通りか

### 2. セキュリティ（OWASP Top 10）
- A03: SQL インジェクション → f-string で SQL を組み立てていないか
- A07: XSS → ユーザー入力をそのまま HTML に埋め込んでいないか
- A08: SSRF → 外部 URL をユーザー入力から構築していないか
- 具体的なコード例: `f"SELECT * FROM t WHERE id = '{user_input}'"` → NG

### 3. パフォーマンス
- N+1 問題: ループ内で個別に SELECT を発行していないか
- joinedload / selectinload が必要な箇所で使われているか
- 不要なクエリが実行されていないか

### 4. エラーハンドリング
- except: pass（例外の握りつぶし）がないか
- 例外発生時に業務コンテキスト（ロット番号等）がログに含まれているか
- DB エラー時のトランザクション処理は適切か

### 5. 型安全性
- 全関数の引数と戻り値に型ヒントがあるか
- Optional の使い方は適切か（None が返る可能性がある場合のみ）

### 6. 命名規約
- 変数名・関数名が snake_case で統一されているか
- 略語（例: ins, insp）を使わず、意味が明確か（例: inspection）
- クラス名が PascalCase で統一されているか

### 7. アーキテクチャ
- Controller 層（api/v1/）にビジネスロジックが混在していないか
- DB アクセスがサービス層を経由しているか
- 循環参照が発生していないか

## 出力形式

### 指摘事項一覧（必ず表形式で出力すること）

| # | ファイル:行 | 重要度 | カテゴリ | 指摘内容 | 修正案 |
|---|-----------|--------|---------|---------|--------|
| 1 | inspection_service.py:73 | Critical | パフォーマンス | _to_response内でループ中にSELECTを発行しN+1問題... | joinedloadで一括ロード... |

重要度の定義:
- **Critical**: セキュリティ脆弱性、データ整合性破壊、性能要件未達。修正必須
- **Warning**: 保守性低下、設計原則違反。修正推奨
- **Info**: コード品質向上の提案。対応任意

### 総合評価（以下の 3 段階で判定）

- **合格（Approve）**: Critical 0 件、Warning 2 件以下
- **修正後マージ可（Approve with changes）**: Critical 1〜2 件
- **差し戻し（Request changes）**: Critical 3 件以上、またはセキュリティ脆弱性あり

### 良い点

レビュー対象コードの良い点（設計通りに実装されている箇所等）も 3 点以上記述すること。

## やってはいけないこと

- 「問題ありません」とだけ回答しないこと（必ず全 7 観点でチェックした結果を記述）
- 重要度を全て Info にしないこと（Critical に該当するものは Critical と判定する）
- 修正案なしに指摘だけしないこと（必ず具体的な修正案を提示する）
- コードを読まずに一般的なアドバイスだけを書かないこと

## 品質チェックリスト（出力前に自己確認すること）

- [ ] 7 つのレビュー観点すべてからチェックしたか
- [ ] 各指摘にファイル名と行番号を明記したか
- [ ] Critical の指摘に具体的な修正案（コード例）を提示したか
- [ ] 総合評価の判定理由を記述したか
- [ ] 良い点も記述したか

@app/services/inspection_service.py
@app/services/alert_service.py
@app/api/v1/inspections.py
@app/api/v1/lots.py
@app/main.py
```

## AI 活用のポイント

| 観点 | 従来の方法 | AI 活用後 | 改善効果 |
|------|-----------|----------|---------|
| 所要時間 | シニアエンジニアが数時間〜1日かけてレビュー | AIが数分で網羅的にチェック | **85%以上の時間削減** |
| 品質 | レビュアーの経験・知識に依存 | 7 つの観点で毎回一定品質 | 属人性を排除 |
| セキュリティ | 脆弱性の見落としリスク | OWASP Top 10 を体系的にチェック | 脆弱性の早期検出 |
| パフォーマンス | N+1問題等は見逃しがち | SQLクエリパターンを自動分析 | 性能問題の早期検出 |

## 人間の役割

AI レビューはあくまで「一次フィルタ」です。人間が行うべきレビューは：

1. **ビジネスロジックの正しさ**: 合否判定のロジックが業務要件と合致するか
2. **設計判断の妥当性**: なぜその実装方法を選んだかの判断
3. **チーム規約との整合**: プロジェクト固有のルールへの準拠
